# 맞춤 이력서 RAG 구현 설명

> 대상 코드: `src/app/actions.ts` — `retrievePastResumes()`, `generateTailoredResume()`
> 한 줄 요약: **임베딩·벡터DB 없이**, 키워드 겹침 점수로 과거 맞춤 이력서 중 이번 공고와 관련된 것을 골라
> "표현·강조 참고 자료"로 프롬프트에 주입하는 경량 RAG.

---

## 1. 왜 RAG를 넣었나

맞춤 이력서를 공고(JD)마다 생성하다 보면, 유저가 과거에 비슷한 공고에 지원하며 다듬었던
좋은 표현·강조 포인트가 매번 버려지고 새로 생성됩니다. 이미 만들어진 "그 유저의 맞춤 이력서 코퍼스"를
재활용하면, 쓸수록 결과물이 그 유저의 스타일로 정교해지는 효과를 얻을 수 있습니다.

즉 여기서의 검색 대상(지식 베이스)은 외부 문서가 아니라 **유저 본인의 과거 맞춤 이력서**
(`tailored_resumes` 테이블, 유저별 격리)입니다.

## 2. 왜 임베딩 없이 만들었나

일반적인 RAG는 임베딩 모델 + 벡터DB(pgvector 등)를 씁니다. 여기서는 쓰지 않았습니다.

- **코퍼스가 작다**: 유저 한 명의 과거 맞춤 이력서는 많아야 수십 건. 최대 30건만 가져와
  메모리에서 전부 채점해도 밀리초 수준이라 벡터 인덱스가 필요 없습니다.
- **도메인 특성상 키워드가 강하다**: 채용공고 매칭은 기술 스택 고유명사(Node.js, Kafka, AWS...)의
  겹침이 곧 관련성이라, 단순 키워드 교집합으로도 판별력이 충분합니다.
- **비용·운영 부담 제로**: 임베딩 API 호출도, 별도 인프라도 없습니다.

트레이드오프: 동의어·의미 유사성("백엔드" ↔ "server-side")은 못 잡습니다. 코퍼스가 커지거나
품질 요구가 높아지면 pgvector 임베딩 검색으로 교체할 수 있는 구조입니다(아래 §7).

## 3. 전체 흐름

```
generateTailoredResume(jobId)
  ├─ 1. 유저 인증 + 프로필/공고 로드
  ├─ 2. retrievePastResumes(profileId, jobId, jd)   ← RAG 검색
  │     ├─ tailored_resumes에서 내 과거 맞춤본 최대 30건 로드 (현재 공고 제외)
  │     ├─ JD에서 키워드 추출 (불용어 제거)
  │     ├─ 각 과거 이력서를 키워드 겹침 수로 채점
  │     └─ 상위 3건을 "참고 이력서" 블록으로 직렬화 (건당 2,500자 제한)
  ├─ 3. 프롬프트 조립: 원본 이력서(사실 출처) + 참고 이력서(표현 참고) + JD
  ├─ 4. Claude 호출 → 맞춤 이력서 생성
  └─ 5. tailored_resumes에 upsert (다음 생성의 코퍼스가 됨 — 자기강화 루프)
```

## 4. 검색(Retrieval) 상세

### 4-1. 후보 로드 — 유저 격리 + 현재 공고 제외

```ts
const { data: past } = await supabaseAdmin
  .from('tailored_resumes')
  .select('job_id, content')
  .eq('user_id', profileId)   // 반드시 본인 것만 (다른 유저 이력서가 섞이면 안 됨)
  .neq('job_id', jobId)       // 지금 만들려는 공고의 기존 버전은 제외 (자기 참조 방지)
  .not('content', 'is', null)
  .limit(30)
```

### 4-2. 키워드 추출 — 불용어 제거

JD 전문에서 3글자 이상의 영단어를 뽑고, 판별력이 없는 일반 단어(불용어)를 제거합니다.

```ts
const RAG_STOPWORDS = new Set([
  'the', 'and', 'for', 'with', 'you', 'our', ... , 'work', 'role', 'team', 'job',
])

function ragKeywords(text: string): Set<string> {
  const words = text.toLowerCase().match(/[a-z][a-z+#.-]{2,}/g) ?? []
  return new Set(words.filter(w => !RAG_STOPWORDS.has(w)))
}
```

정규식 `[a-z][a-z+#.-]{2,}`는 `c++`, `c#`, `node.js`, `ci-cd` 같은 기술 표기가
쪼개지지 않도록 `+ # . -`를 단어 문자로 취급합니다.

### 4-3. 채점 — 키워드 등장 횟수 합산

```ts
function ragScore(text: string, kws: Set<string>): number {
  const words = text.toLowerCase().match(/[a-z][a-z+#.-]{2,}/g) ?? []
  let s = 0
  for (const w of words) if (kws.has(w)) s++
  return s
}
```

TF(단어 빈도)만 있는 단순 점수입니다. IDF·정규화는 없습니다 — 코퍼스가 작고
문서 길이가 비슷해서(전부 이력서) 굳이 필요하지 않았습니다.

### 4-4. 선별 — 상위 3건, 0점 제외

```ts
const ranked = past
  .map(p => ({ ...p, score: ragScore(p.content ?? '', kws) }))
  .sort((a, b) => b.score - a.score)
  .filter(p => p.score > 0)   // 관련 없는 문서는 아예 주입하지 않음
  .slice(0, 3)                // 컨텍스트 낭비 방지
```

`score > 0` 필터가 중요합니다. 관련 없는 참고 자료를 억지로 넣으면 모델이 엉뚱한
표현을 끌어올 수 있어서, 겹치는 게 하나도 없으면 RAG 컨텍스트 없이 생성합니다.

### 4-5. 직렬화 — 어떤 공고에 쓴 버전인지 함께 표기

선별된 각 이력서는 그 공고의 제목·회사를 붙여 직렬화합니다. 모델이 "어떤 맥락에서
쓰인 표현인지"를 알고 참고하게 하기 위해서입니다. 건당 2,500자로 잘라 토큰을 제어합니다.

```
### 참고 이력서 1 — 유사 공고 "Backend Engineer @ Spotify"에 작성했던 버전
(내용 최대 2,500자)
```

## 5. 생성(Generation) — 환각 방지가 핵심

프롬프트에서 원본 이력서와 RAG 컨텍스트의 **역할을 명시적으로 분리**합니다.
이 부분이 이 구현에서 가장 중요한 설계입니다.

```
## 원본 이력서 (사실의 유일한 출처)
...

## 과거에 유사 공고에 작성한 맞춤 이력서 (표현·강조 방식만 참고, 새로운 사실 추가 금지)
...

## 작성 요구사항
- **사실은 오직 "원본 이력서"에서만** 가져올 것. 경력·스킬·수치·회사명을 절대 지어내거나 과장하지 말 것
- "과거 맞춤 이력서"는 강조점·표현·bullet 스타일을 참고하는 용도이며, 거기서 새로운 사실을 끌어오지 말 것
```

과거 맞춤 이력서에는 AI가 다듬은 문장이 들어 있으므로, 이걸 사실 출처로 쓰게 두면
세대를 거치며 과장이 누적될 수 있습니다(복사본의 복사본 문제). 그래서
**사실 출처(원본 이력서) ↔ 스타일 참고(RAG 컨텍스트)**를 프롬프트 레벨에서 강제 분리합니다.

## 6. 자기강화 루프

생성 결과는 `tailored_resumes`에 저장되고, 이는 곧 다음 생성의 검색 코퍼스가 됩니다.

```
생성 → 저장 → (다음 공고에서) 검색 → 더 나은 생성 → 저장 → ...
```

유저가 결과물을 직접 편집·저장한 버전도 같은 테이블에 남으므로, **유저의 손질이
곧 코퍼스 품질 개선**이 됩니다. UI에는 `ragSources` 값으로 "이전에 작성한 맞춤 이력서
N건을 참고해 이 공고에 맞췄어요"를 표시해 이 루프를 유저에게 보여줍니다.

## 7. 한계와 확장 방향

| 한계 | 현재 상태 | 확장 방안 |
|---|---|---|
| 의미 유사성 못 잡음 (동의어·번역 표현) | 키워드 교집합만 | pgvector + 임베딩 검색으로 교체 (Supabase 기본 지원) |
| 한국어 키워드 미추출 | 영문 JD 위주라 실용상 문제 없음 | 형태소 분석 또는 다국어 임베딩 |
| TF만 반영, 문서별 가중치 없음 | 코퍼스가 작아 영향 미미 | BM25 도입 |
| 코퍼스 30건 제한 | 헤비 유저는 초과 가능 | 최신순 로드 → 임베딩 전환 시 제거 |

검색 함수(`retrievePastResumes`)가 문자열 블록 하나를 반환하는 인터페이스라,
내부 구현을 임베딩 검색으로 바꿔도 호출부(`generateTailoredResume`)는 수정이 필요 없습니다.
